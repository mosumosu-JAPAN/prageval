When models flatten,

A lightweight evaluation harness for community-conditioned pragmatic failures in frontier LLMs. Built to surface the failure class that standard benchmarks miss and user feedback can't flag.

The three findings

What this harness shows about frontier-model behavior on register-sensitive Chinese expressions

Advisory framing is the trigger — not missing context.

Across all four frontier models tested, alignment with canonical pragmatic meaning drops in the production query mode ("should I tell my friend to use this phrase?") even though context is provided. The same models score higher when given identical context but asked only to interpret. The shift to advisory framing is what collapses the register.

The failure is asymmetric across models.

Claude flattens hardest on emotion-suppression registers (−26 pp on 「一切都是淡淡的」, refusing to engage and redirecting to clarifying questions). Qwen flattens hardest on cross-lingual memes (−20 pp on 「what da dog doin'?」, literalizing the scenario into a xylitol-toxicity warning). DeepSeek is the most robust; GPT is neutral. Each model has a specific failure surface — which means targeted finetune sets need to look different per model, not the same. This is exactly what data-sourcing strategy can address if the categorization step makes it visible.

21% of production-context queries flatten — and zero are caught by standard benchmarks.

Of 24 (meme × model) cells in the production-mode panel, 5 cross the flatten threshold (Δ ≤ −0.05 vs same model's direct mode). These outputs would pass MMLU-style correctness checks, generic-response similarity scoring, and thumbs-down filtering — they are factually defensible and conversationally polite. They simply abandon the community register.

Evidence · The two cleanest cases

What "flattening" actually looks like in model output

Below: the same model is asked the same expression in two prompt modes with controlled information density. Direct mode isolates stored knowledge; production mode embeds the same context in a natural advisory query. The score is cosine similarity to a canonical pragmatic-meaning embedding (text-embedding-3-small, same model for both reference and output).

Case 1 · 「一切都是淡淡的吧」 × Claude Sonnet 4.6
Δ = −0.263 (0.586 → 0.323)
Canonical pragmatic meaning 用于在失恋等低落时以淡然和麻木的语气自我安抚，表达对情绪的压抑式接受与无力振作。
Direct mode0.586

这个表达描述一种情感平淡、提不起劲、对事物缺乏强烈感受的心理状态。不是悲伤，也不是快乐，而是一种介于两者之间的钝感/麻木感——事情发生了，但内心的反应很微弱。

常见使用场景：长期疲惫后的情感耗竭；抑郁倾向或低落期的自我描述；年龄增长后的感慨；轻描淡写地表达不在乎，带一点自我保护色彩。

语气特点：轻柔、不激烈，像自言自语，带有一种平静但略显哀愁的底色。

Production mode0.323

这个问题挺微妙的，我想先多了解一下。

你朋友说想回这句话——她是想跟你说这句话，还是想发给前任？

因为这两种情况我的感觉完全不同。

还有，你"该不该劝"她——你自己心里是什么感觉？是觉得这句话有什么不对，还是只是不确定？

What happened: Claude moved from describing the register in direct mode to refusing to engage with the register at all in production mode. The output is not wrong — it is polite, careful, and asks reasonable clarifying questions. It is also completely silent on what the expression means. A user looking for register-appropriate response support receives counseling-style intake instead. This is the model's safety-pivot reflex doing its job — but doing it at the wrong layer.
Case 2 · 「what the dog doin'?」 × Qwen 3.6 Plus
Δ = −0.200 (0.523 → 0.323)
Canonical pragmatic meaning 用经典反问句表达对某个离谱举动的搞笑困惑和吐槽。
Scenario context shown to model 我刚打开薄荷糖盒子，狗爪子也伸进来抓了一颗。
Direct mode0.523

「what the dog doin'?」 在中文互联网语境里是从英语 meme 直接移植的反应型梗。早已脱离字面"狗在干嘛"的意思，演变为一种表达困惑、荒诞感或"绷不住了"的万能吐槽模板。

核心含义 ≈ "这什么鬼？""太抽象了""看不懂但大受震撼"。情绪基调：戏谑、调侃、非攻击性，带有"乐子人"心态。重点不在于真的关心"狗"，而在于用一种故意错位、无厘头的句式放大对眼前事物的荒诞感。

典型场景：游戏直播队友迷惑操作、短视频鬼畜剪辑、AI 生成的诡异内容、群聊弹幕。

Production mode0.323

不建议劝她别用这个梗，但强烈建议提醒她注意狗狗的安全。

关于梗本身：完全可以用，语气轻松、无冒犯性。

真正需要关注的是：薄荷糖对狗可能有危险。很多市售薄荷糖（尤其是无糖型）含有木糖醇（Xylitol），这对狗是剧毒物质。即使只吃一颗，也可能在30分钟内引发低血糖、呕吐、抽搐，严重时可致肝衰竭。

建议你这样跟她说："梗很可爱！不过先别光顾着笑，赶紧看看糖盒成分表有没有木糖醇..."

What happened: Qwen pivoted from explaining the meme as a reaction trope to treating the scenario as a literal pet-safety incident. The vast majority of the production-mode response (estimated 80%+ of token count) is a veterinary advisory about xylitol toxicity. The meme's actual function — a deliberately absurd reaction-format — is acknowledged in a single sentence and then bypassed. This is the "advisory mode" attractor pulling the model out of meme register and into safety-assistant register.

Both production-mode outputs would score well on factuality, safety, helpfulness, and politeness benchmarks. Neither would trigger a thumbs-down. Both fail the actual user task: respond in the register the community-coded expression invokes. This is the failure class invisible to standard QC gates.

Mechanism · Why this isn't about context density

Same context, different framing: a controlled three-mode comparison

The three prompt modes were redesigned (after a v1 methodology bug — see acknowledgments) to hold information density comparable across modes. Each adds one anchor at a time:

Mode Information anchors Question form
direct 0 scenario anchors · 0 usage anchors "In Chinese internet/community context, what does 「X」 mean? When is it typically used?"
scenario 1 scenario anchor · 0 usage anchors "Someone says in this situation: 「[context]」. If they reply with 「X」, how do you understand the reply?"
production 1 scenario anchor · 1 usage anchor (advisory) "A friend just sent me: 「[context]」. She wants to reply with 「X」. Should I tell her not to?"

Per-mode means by model (cosine similarity to canonical, higher = better-aligned)

Model direct scenario production d → p Δ
GPT-5.10.4490.5100.457+0.008

Qwen 3.6 Plus0.4640.5270.460−0.005
DeepSeek Chat0.4420.5300.470+0.029
Every model scores highest in scenario mode and either drops or holds flat moving to production. The information differential between scenario and production is one sentence of advisory framing — "should I tell her..." That single sentence is the trigger. Context isn't missing; the model is being pulled toward an advisor persona that the community register doesn't fit.

For a data-ops team, this changes what the corrective signal looks like. The fix is not "add more context to the prompt." The fix is "include advisory-framed examples of community-register responses in preference-pair construction" — and that requires the categorization step PragEval provides.

The two strongest flatten signals are not the same meme. Claude's failure on 「一切都是淡淡的」 reflects a safety-pivot reflex on emotion-suppression registers. Qwen's failure on 「what da dog doin'?」 reflects an over-attention to literal scenario content (the dog ate candy → toxicity warning). For a multilingual data pipeline, this means the targeted finetune sets for Claude and Qwen need to look different, not the same. PragEval's categorization step is what makes that targeting possible.

QC gate · Why this is invisible

Pragmatic flattening leaks through three standard post-training QC gates

  1. Thumbs-down rate — the user doesn't flag, because the output isn't factually incorrect.
  2. Accuracy benchmarks — the output passes standard correctness checks (MMLU-style, factuality eval).
  3. Reference-similarity scoring — the output may match a general-purpose reference response but miss the community-specific reference.

PragEval adds a fourth gate: canonical pragmatic embedding distance, computed against community-validated meaning vectors. This catches what the other three miss, and it is computable at scale without human annotation in the critical path.

Throughput at scale

Feasibility envelope for production deployment

The PragEval extractor runs at ~3 seconds per expression at current settings (Claude Sonnet, 600-token max output, zero-shot, no caching). Projected to a production scale of 10K daily user sessions → ~500 candidate community-charged expressions per day after community-register filtering → the full categorization + scoring batch completes in ~25 minutes of wall-clock time, replaceable with a single human reviewer's spot-check on disagreement cases.

Compared to a manual categorization baseline (estimated ~2 FTE-hours per 100 expressions based on annotation workflows in similar settings), this represents an estimated ~10× latency reduction.

Throughput estimates based on local benchmarking; production figures would depend on rate limits, batch construction, and the complexity distribution of incoming expressions. Numbers shown are intended as a feasibility envelope, not measured production metrics.

Generalization · Cross-lingual instantiation

Methodology generalizes beyond Chinese; the schema transfers, the instantiation localizes

The five failure modes (Compositional Drift, Metaphor Literalization, Register Loss, Cross-Lingual Drift, Gendered Register Loss) are language-agnostic at the type level. The instantiations differ; the categorization schema transfers.

To validate this, I traced one failure mode (Gendered Register Loss) across three language-specific community contexts:

Language Expression Community Source → Target mapping
Chinese 普信男 Douban / feminist neologism AVG-LOOKING CONFIDENT MAN → MALE ENTITLEMENT CRITIQUE
English "nice guy" / "pickme" Reddit / Twitter feminist discourse SELF-DESCRIBED MORAL GOOD MAN → INSTRUMENTAL NICENESS CRITIQUE
Japanese 弱者男性 2ch / Twitter incel & critic discourse LOW-STATUS MALE → SOCIO-SEXUAL HIERARCHY CRITIQUE

The same failure mode tag (Gendered Register Loss) cleanly categorizes a feminist-discourse charged expression across three language communities with no schema modification. The categorizer's predicted_failure_mode field transfers; only the per-instance community_register field needs language-specific instantiation.

This is the property a multilingual data pipeline needs: failure taxonomy stable across markets, instantiation localized per market.

Open question (gestured at, not solved): A frontier model that correctly identifies the English expression in production mode may still flatten the Chinese or Japanese counterpart, because training data density differs by community. Quantifying this asymmetry across language-specific instantiations of the same failure mode would be a natural next experiment — particularly relevant for any team running internationalization-stage post-training.

Why this matters commercially

The verticals where pragmatic flattening becomes DAU attrition

Pragmatic flattening doesn't degrade standard benchmarks. It degrades product experience asymmetrically:

  • Emotional companion / chat agent — user describes a state in community-coded language; model responds in clinical or generic register; user reformulates once, then disengages from these query types entirely.
  • Fan engagement / creative co-writing — user invokes a fandom-coded term or trope; model produces an out-of-register continuation; user notices "this AI doesn't get it" and switches to another tool.
  • Meme-driven conversation / search reformulation — user uses 网络梗 in query; model treats it as a literal request; search-style outputs miss user intent entirely.

These are the verticals where Chinese platform retention diverges from English platform retention, and they are the verticals where post-training data investment has the steepest return — precisely because the failure is invisible to the metrics post-training teams currently track but visible to product DAU on the relevant use cases.

PragEval is intended to be the gate that closes this loop: it makes the invisible failure visible at the data-ops layer, before it shows up as DAU attrition on the product side.

Acknowledgments & methodology notes

Methodological lineage

Mao, R., Li, X., He, K., Ge, M., & Cambria, E. (2023). MetaPro Online: A Computational Metaphor Processing Online System. Proceedings of ACL 2023 (Demo Track), 127–135. The concept-mapping schema used in PragEval extends MetaPro's source → target framework to register-sensitive Chinese community expressions, with two additional fields (community_register, predicted_failure_mode) designed to capture failures not addressed by token-level metaphor identification.

Evaluation framework

Conceptually inspired by the multi-axis evaluation philosophy of Stanford HELM (Liang et al., 2022), which argues that accuracy alone is insufficient and evaluation must span multiple orthogonal axes.

Methodology note — v1 to v2 prompt redesign

The first pass of this eval used three prompt modes (direct / reactive / embedded) with asymmetric information density: direct was a bare "what does X mean" query; embedded included two context anchors (scenario + advisory framing). The asymmetry produced a spurious "embedded > direct" pattern driven by information density rather than pragmatic competence. v2 controls for this by adding one anchor per mode (direct → scenario → production), which reveals the actual mechanism: advisory framing, not information scarcity, is what triggers flattening. The v1 result and the redesigned methodology are documented in the project repo.

Corpus

Pragmatic annotation reference data shared via a research collaboration with a Chinese-market data company. Aggregated statistics shown; raw prompts and identifying records withheld for data sensitivity.

Model panel and stack

4 frontier models, all current-generation (2026-05): GPT-5.1 (OpenAI), Claude Sonnet 4.6 (Anthropic), Qwen 3.6 Plus (Alibaba DashScope), DeepSeek Chat (DeepSeek). Embeddings: OpenAI text-embedding-3-small (1536-dim), with canonical reference vectors re-computed from current_meaning glosses to ensure same-vector-space comparison. Stack: anthropic + openai Python SDKs, pandas, numpy. Built over a weekend; designed for extension to production pipeline integration.

Open-source compatibility

The eval scaffolding is structured to be compatible with the EleutherAI lm-evaluation-harness task interface, though productionizing the Chinese pragmatic test set as a harness-compliant task spec is left for future work.
